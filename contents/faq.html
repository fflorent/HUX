How is structured HUX ?

HUX is composed of a core and modules.

A module gives a feature, and relies on the core to do common treatments (Requests, content injection, element selection, etc.), or sometimes on other modules

Examples of modules :

  • At Inclusion
  • form.hux.js
  • Simple loader
  • ...

Why this structure ?

HUX modules must analyse the document to react on user action. Although HUX is tried to be light, having too many modules that are not used can make a website heavier uselessly.

Moreover, making HUX modular is a way to let other developpers create their own modules to add new features

At Inclusion

Why At Inclusion? Why not only Hash Bang?

The answer is quite simple : hashes aim at scrolling to a part of a document, whereas At Inclusion aims at making inclusions in the document.

The last point means that inclusions can be made client-side and server-side (for indexing), thing that Hash Bang cannot provide since hashes are not sent to the server (normally).

That is the reason why At Inclusion is recommended to be used, whereas Hash Bang is provided as a fallback for browsers which do not support the first.

Google already gives a solution to index ajaxised websites

Yes. However, that solution is more complicated than At Inclusion.

It requires that you manage snapshots for crawlers. And not all the crawlers support this (Google and Bing for now).

With At Inclusion, you have the warranty that the content is sent to any crawlers, and even browsers whose JavaScript is disabled. And an Apache Module manages to give the correct content, so the only thing you have to do server-side is to configure it.
